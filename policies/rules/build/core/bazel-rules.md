Runbook Targets in BUILD.bazel
Ansible playbooks, scripts, and notebook executions can also be Bazel targets. This makes running any operational procedure consistent with running any other package command.

runbooks/ansible/BUILD.bazel
│
├── sh_binary target: deploy-payment-service
│   └── runs ansible-playbook playbooks/services/payment-service/deploy.yml
│       with the correct inventory for DEPLOY_ENV
│
├── sh_binary target: rollback-payment-service
│   └── runs ansible-playbook playbooks/services/payment-service/rollback.yml
│
└── Usage
    └── bazel run //runbooks/ansible:deploy-payment-service -- --env=staging
 
runbooks/scripts/BUILD.bazel
│
├── py_binary target: fix-orphaned-workflows
│   └── runs scripts/workers/temporal-payment-worker/fix-orphaned-workflows.py
│
└── sh_binary target: inspect-dead-letter-queue
    └── runs scripts/workers/queue-email-worker/inspect-dead-letter-queue.sh


Bazelisk — Version Management
Bazelisk is the version manager for Bazel. It reads .bazelversion and downloads the correct Bazel binary automatically. Every engineer and every CI runner uses Bazelisk, never raw Bazel.
Install Bazelisk once globally — it handles all Bazel versions automatically.
.bazelversion contains a single version string such as 7.4.1.
Updating Bazel for the whole team means changing one line in .bazelversion and committing it.
Bazelisk is available via npm, Homebrew, and direct binary download.
CI installs Bazelisk as the first step in every pipeline.

Migration from Per-Language Tools to Bazel
Start with one language — the one with the most packages or the slowest builds.
Add MODULE.bazel and .bazelrc to the project root.
Write BUILD.bazel for the first package in the chosen language. Use the standard target set.
Run bazel build //packages/{lang}/{name}/... and verify it builds correctly.
Run bazel test //packages/{lang}/{name}/... and verify all tests pass.
Add the second language. Add its ruleset to MODULE.bazel. Write BUILD.bazel for one package.
Enable disk cache first (zero infrastructure required). Measure the speedup.
Migrate remaining packages one at a time. Keep old per-language scripts working during migration.
Set up remote HTTP cache once two or more languages are migrated.
Replace per-language CI steps with unified bazel build //... and bazel test //... steps.
Add container image and deploy targets once the build and test pipeline is stable.

Non-Negotiables
.bazelversion is committed and pinned. Every engineer and CI uses Bazelisk.
No target depends on anything outside the Bazel dependency graph — no implicit file reads, no curl in BUILD files.
All external dependencies are declared in MODULE.bazel and locked. No unpinned dependencies.
BUILD.bazel files are formatted with buildifier before every commit. Zero formatting violations.
Go BUILD.bazel files are generated by Gazelle. They are never hand-edited.
bazel build //... and bazel test //... pass on every commit in CI.
Remote cache is enabled in CI from stage 2 onwards. Uncached builds in CI at scale are a performance violation.
No package uses its language-native build tool (cargo, gradle, npm run build) as the primary build in CI after Bazel migration — Bazel is the single build surface.
